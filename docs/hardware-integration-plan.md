# 硬件配合落地计划（Phase 6 的展开）

> 路线前提见 [`architecture-decision-omni-vs-xiaozhi.md`](architecture-decision-omni-vs-xiaozhi.md)：走 Qwen-Omni 实时（B），设备侧第一档。本文把"怎么和硬件配合"拆成可执行步骤。

## 核心：拆开"大脑"与"身体"

现在是**电脑全包**——Electron 渲染层当身体（麦克风/喇叭/表情），主进程当大脑（Omni + 记忆）。与硬件配合 = **把主进程的大脑抽成独立服务器，身体换成 ESP32**。`src/core` 两边复用，不重写。

设备说的是**小智协议**，大脑说的是 **Omni**，所以服务器中间要加一层**翻译**——这是最大的一块新活，也是最不确定的一环。

```
ESP32 设备(身体)            我们的服务器(大脑)
 麦克风/喇叭/屏/唤醒  ──小智WS协议──▶  翻译层 ──▶ Qwen-Omni 实时 + 记忆
                     ◀──音频/情绪──        （src/core 复用）
```

## 步骤（按依赖顺序）

1. **抽出独立服务器**：把大脑（`src/main` 的会话编排 + `src/core` 的 Omni 内核/记忆）从 Electron 里剥离成无界面 Node 服务器，可独立运行。
2. **装设备接口 = 翻译层**（最核心、最不确定）：实现小智 WebSocket 协议——设备来的音频帧 → 喂 Omni；Omni 回的音频 → 按小智协议发回。需先研究小智协议（`../xiaozhi-server/` 留作参考）。
3. **假设备验证（不用硬件）**：用 [py-xiaozhi](https://github.com/huangjunsen0406/py-xiaozhi)（PC 客户端）当假设备连本服务器，完整对话一轮。提前验证翻译层 + 回合制 Omni 是否顺，风险在买硬件前暴露。
4. **买硬件**：现成 ESP32-S3 成品板（带麦克风阵列更利于回声消除），不焊电路。
5. **fork 固件**：只改两处——唤醒词（如 "YUI~"）、服务器地址指向本服务器。烧录。
6. **真机调试**：设备连服务器对话，调回声（硬件 AEC）、调延迟。
7. **表情上屏**：把情绪→表情（Phase 4）搬到设备屏幕或外接屏。

## 复用 vs 新写

| 复用（已做好） | 新写 |
|---|---|
| `src/core` Omni 内核、情绪推断、断句 | 无界面服务器外壳 |
| 记忆机制、人格、配置校验 | **小智↔Omni 翻译层**（最大块） |
| | 固件 fork（唤醒词 + 服务器地址） |

## 关键判断

- **最该先做、风险最高的是第 2~3 步**（协议翻译 + 假设备验证），**完全不需要硬件**。做通了才证明"B + 第一档"走得通——硬件未到先啃这个。
- **唤醒词住在固件里**，改它必须 fork 固件重烧，A/B 两条路都一样，不是 B 的额外负担。
- **服务器部署**：家里常开的小主机/树莓派，或云服务器；设备开机连同一地址。

## 现在（无硬件）就能做

- 抽独立服务器（第 1 步，重构，复用现有代码）。
- 研究小智协议 + 起草翻译层设计（第 2 步前置）。
- py-xiaozhi 假设备链路（第 3 步），验证翻译层。
- 与硬件无关的打磨：表情 UI（Phase 4）、回声/延迟调参、人格固化。
